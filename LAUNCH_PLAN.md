# خطة تشغيل ونشر بابل AI

## الحالة الحالية

التطبيق يعمل محليا كنسخة MVP:

- الاسم: بابل AI
- الرابط المحلي: http://127.0.0.1:5174/
- كود الوصول الحالي: arkan2026
- النموذج المضبوط: gpt-5.4-nano
- المفتاح محفوظ محليا في `.env.local`
- يوجد حد يومي عبر `DAILY_REQUEST_LIMIT`

## العائق الحالي

الواجهة والخادم وكود الوصول تعمل. عند إرسال طلب فعلي، OpenAI ترجع خطأ:

```text
Usage, rate, or billing limit reached.
```

المعنى العملي: حساب OpenAI API يحتاج رصيد API متاح أو رفع/تفعيل حدود الاستخدام.

## قبل النشر العام

لا ننشر التطبيق للعامة قبل هذه النقاط:

1. تفعيل رصيد OpenAI API والتأكد من أن الطلبات ترد.
2. تدوير مفتاح OpenAI لأن جزءا منه ظهر سابقا في صور/أسماء ملفات.
3. تحويل الخادم من PowerShell محلي إلى Backend مناسب للاستضافة.
4. إضافة نظام مستخدمين أو أكواد متعددة.
5. وضع حد استخدام لكل مستخدم.
6. إضافة صفحة سياسة خصوصية وشروط استخدام.
7. إضافة لوحة مالك بسيطة لمراقبة الاستخدام بدون تخزين أسئلة المستخدمين.

## خطة الموقع فقط

### المرحلة 1: تشغيل محلي مستقر

- التأكد من رصيد OpenAI API.
- اختبار رسالة قصيرة بنجاح.
- تغيير كود الوصول إلى كود قوي.
- ضبط `DAILY_REQUEST_LIMIT` حسب الميزانية.

### المرحلة 2: نسخة موقع قابلة للاستضافة

- بناء Backend مناسب للنشر.
- نقل الواجهة الحالية كما هي مع تحسين بسيط للجوال.
- نقل أسرار التشغيل إلى متغيرات بيئة في منصة الاستضافة.
- عدم كشف مفتاح OpenAI في المتصفح.

### المرحلة 3: تحكم المستخدمين

- أكواد دخول متعددة أو حسابات بسيطة.
- حد يومي لكل كود.
- إمكانية إيقاف كود معين.
- عداد استخدام بدون تخزين نص الأسئلة.

### المرحلة 4: النشر

- استضافة مبدئية على Render أو Railway.
- ربط دومين اختياري.
- اختبار خاص قبل فتحه لأي مستخدمين.

## تقدير أولي للتكلفة الشهرية

- استضافة بسيطة: 5 إلى 10 دولار شهريا.
- دومين اختياري: 10 إلى 20 دولار سنويا.
- OpenAI API: حسب الاستخدام، ويفضل البدء بحدود صارمة.

الانطلاقة المناسبة: موقع خاص أو شبه عام بكود وصول، وليس نشر مفتوح بالكامل.
